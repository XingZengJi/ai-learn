# 拆书工坊 · 实践课方案

> 定位：Claude 101（c01）+ Claude Code 101（c02）的综合实战项目。测验是实验课，这里是实践课。
> 制定日期：2026-07-12
> 玩法：**你开车，Claude 当教练**——所有「知识点时刻」必须你亲手操作，教练只给脚本、讲解和检查点。
> 成果绑定：完成后点亮成就地图 p2 行星「拆书工坊 · 小说拆解流水线」。

---

## 一、项目蓝图

把你现有的 `story-hook-analyzer` 个人技能，升级成一条完整的小说拆解流水线：

```
texts/（小说原文，只读圣地）
   ↓  拆解 Skill（你的编辑 SOP 固化成项目资产）
   ↓  子智能体并行处理多篇，只回摘要
reports/（结构化拆解报告：基础信息 / 核心梗五维 / 段落大纲表 / 钩子评估）
   ↓  Hooks 强制校验格式、保护原文
index.html（报告库网页，零构建纯静态，可上 GitHub Pages）
```

做完之后你得到：一个真会持续用的拆书工具 + 一次「从空目录到上线」的完整工程经历。

## 二、教练规则（贴在心上）

1. **必须你亲手做的**：写每一条提示词、Shift+Tab 切模式、跑 `/init` `/agents` `/hooks` `/compact` `/clear` `/context` `/mcp`、审阅和批准计划、git 首次建仓。教练可以给你「建议提示词模板」，但你要理解后自己改写再发出，禁止原样复制。
2. **教练做的**：每阶段开场讲清「今天用哪些知识点、为什么」；你卡住时先提示课次笔记位置让你自己翻，翻完还不会才直接讲；阶段结尾带你过「知识点回收清单」。
3. **卡住 ≠ 失败**：卡住的地方就是测验里你答对了但其实没懂的地方——这正是实践课要找的东西，记进复盘。
4. 每阶段结束：commit（用你的风格写中文提交信息）+ 花 5 分钟向教练**口头讲一遍今天每个知识点是干嘛的**（费曼检查）。讲不清的标记出来。

## 四、五阶段教案

### 阶段 1 建仓与裸跑（约 1.5 小时）

**学习目标**：亲历「没有 CLAUDE.md 的 Claude 每次都失忆」，再用 /init 治好它；用三要素写提示词并观察智能体循环。

**你的操作清单**：

1. 建仓（第一课就是这个，别让教练代劳）：
   ```bash
   mkdir -p ~/workspace/projects/book-workshop && cd ~/workspace/projects/book-workshop
   git init
   mkdir texts reports
   ```
2. 挑 1 篇你熟悉的短篇小说存进 `texts/`（你人工拆解过的最好，阶段 4 做 evals 要用）
3. 在仓库根目录启动 `claude`——注意：在哪启动，它就只能访问哪（k0202）
4. **不写 CLAUDE.md，直接裸跑**。用三要素句式写第一条提示词，模板（理解后自己改写）：
   > 【场景】我是网文编辑，正在搭建一个小说拆解工作流，texts/ 下有一篇样例。【任务】请阅读它，按「基础信息→核心梗→段落大纲→付费钩子评估」四步输出一份拆解分析。【规则】核心梗每个维度 1–2 句话，段落大纲用表格，判断不出的标「暂不明显」不要硬编。
5. 观察它的循环：收集上下文（读文件）→ 行动 → 核查（k0201）；注意每次权限询问时它想干什么再批准
6. 迭代 2–3 轮修输出（「第二部分太啰嗦」「表格加一列」），**把每个你需要纠正的点记在纸上**
7. 会话尾声跑 `/init`，打开生成的 CLAUDE.md，把纸上记的纠正点亲手写进去（第 8 课「让子弹飞一会儿」的原版实践，k0206）
8. 首次提交：让 Claude 生成中文提交信息，你审阅后批准

**教练检查点**：提示词是否真有三要素｜权限询问你是否看懂了它要干嘛才批的｜CLAUDE.md 里是不是只写了「真被纠正过的点」而不是空泛规则

**知识点回收清单**：
- [ ] 能说清 Claude Code 和 Claude.ai 的本质区别（k0201）
- [ ] 能说清「为什么建议先裸跑再写 CLAUDE.md」（k0206）
- [ ] 我的提示词三要素各是什么（c01）

### 阶段 2 完整工作流做核心功能（约 1.5 小时）

**学习目标**：把「探索→计划→编码→提交」走成肌肉记忆；三种权限模式都亲手切过。

**功能需求**（你以产品负责人身份提出）：报告产线 v1——固定报告结构（沿用四步 SOP）、报告落盘 `reports/<书名>.md`、统一命名规范。

**你的操作清单**：

1. 开工前跑一次 `/context`，记下基线占用（k0204 伏笔）
2. Shift+Tab 切到 **Plan Mode**（确认输入框下方出现 Plan Mode 标记）
3. 用第 5 课「WebP 转换」的句式写需求：说清目标 + 让它判断「该怎么做、要不要新增文件、报告结构怎么定」
4. 审阅计划，**至少提出 1 处修改**再批准——体验「代码写出来之前是纠偏成本最低的时刻」（k0203）
5. 执行阶段：前半程用默认审批模式逐个批，后半程 Shift+Tab 切自动接受模式，体会两者差别（k0202）
6. 完成后再跑 `/context`，和基线对比：空间被什么吃掉了
7. 用产线拆 1 篇新文本验证可重复性
8. 提交：让 Claude 按你的风格写提交信息

**教练检查点**：计划审阅是不是走过场（提不出修改点=没认真读）｜你能否说出自动接受模式下命令是否还需要批准（需要）

**知识点回收清单**：
- [ ] 四步工作流每步的目的（k0203）
- [ ] 三种权限模式分别什么时候用（k0202）
- [ ] /context 里占大头的是什么（k0204）

### 阶段 3 上下文管理与子智能体（约 1.5 小时）

**学习目标**：让「独立上下文」从名词变成体感；提交前审查成为习惯。

**你的操作清单**：

1. 跑 `/agents` → Create new agent，亲手创建「报告审校员」：作用范围选本项目、用途写清「按拆解报告规范检查 reports/ 下的报告质量并列出问题清单」、**工具只勾只读**（审查员标问题、不改文件，k0205）
2. 把审校员的配置文件提交进仓库（团队共用同一个审查员的做法）
3. 投 2–3 篇新文本，提示词里明确要求「用子智能体并行拆解，各自独立处理」——观察主上下文只收到摘要、没有被读文件过程塞爆（k0207；对比阶段 1 裸跑的体感）
4. 上下文实操（k0204）：
   - 拆完一篇还要继续拆下一篇 → 跑 `/compact`，看压缩摘要长什么样
   - 全部拆完要换「整理报告索引」这个新任务 → 跑 `/clear`
   - **口头向教练解释这两次分别为什么选这个命令**（费曼检查点）
5. 提交前让审校员过一遍所有新报告，按它的问题清单修完再 commit

**教练检查点**：审校员是否真的只读｜/compact 与 /clear 的选择理由能否说清｜并行拆解时你是否观察了主上下文的干净程度

**知识点回收清单**：
- [ ] 子智能体的核心价值一句话（k0207）
- [ ] /compact vs /clear 的选择原则（k0204）
- [ ] 为什么审查要用「新眼睛」（k0205）

### 阶段 4 Skill 化 + Hooks 硬规则（约 2 小时）

**学习目标**：把个人经验变成项目资产（Skill）；分清「拜托模型」和「强制卡点」（Hook）。

**你的操作清单**：

1. **Skill 升级**（k0208）：和 Claude 对话，把 `~/.claude/skills/story-hook-analyzer` 升级成本项目的项目级 Skill——在原四步 SOP 基础上增加：报告必须落盘到 `reports/`、文件命名规范、报告头部元信息（拆解日期/字数/频道）。让 Claude 生成后你亲手读一遍 SKILL.md，确认描述里写清了触发时机
2. **轻量 evals**（c01 第 3 课四步法）：拿出你人工拆解过的 2 篇文 → 让新 Skill 跑一遍 → 逐维度对比 Claude 版和你的人工版（核心梗准不准、卖点句有没有钩子感）→ 把差距写成 Skill 提示词的修改，再跑一遍看是否收敛
3. **Hook 一：格式守卫**（k0210）：先让 Claude 写一个校验脚本（检查报告四大节齐全、大纲表存在），然后你亲手跑 `/hooks` 配置 PostToolUse、匹配器 `Write|Edit`——以后任何报告写入都自动校验，不合格立刻知道
4. **Hook 二：原文保护**（k0210）：配置 PreToolUse，拦截一切对 `texts/` 的写操作，退出码 2 + stderr 写明「原文只读，拆解不许改原文」。**故意让 Claude 去改一篇原文，亲眼看它被拦下、读到原因、然后调整做法**
5. 把 `.claude/settings.json` 提交进仓库（团队共享卡点）

**教练检查点**：能否说清这两条规则为什么用 Hook 而不是写进 CLAUDE.md（确定性 vs 大概率遵守）｜evals 前后 Skill 输出是否真的变好了

**知识点回收清单**：
- [ ] Skill 与 CLAUDE.md 的分工（k0208）
- [ ] PreToolUse 退出码 0 / 2 / 其他的区别（k0210）
- [ ] evals 四步是哪四步（c01）

### 阶段 5 MCP + 报告库网页 + 总装复盘（约 2 小时）

**学习目标**：看懂 MCP 的价值和代价；独立二刷完整工作流，检验是否真学会。

**你的操作清单**：

1. **解剖 MCP**（k0209）：跑 `/mcp` 看已装的 context7——它有哪些工具、状态如何；跑 `/context` 看它的工具定义占了多少上下文；口头回答教练：「什么情况下你会禁用它？」
2. 把一个 MCP 服务器配置成 **Project 范围**（`.mcp.json` 提交进仓库），说清它和 Local/User 范围的区别
3. **独立二刷**（本阶段核心）：报告库网页——纯静态 `index.html` 列出 reports/ 全部报告（参考成就地图的零构建做法）。**这次教练全程闭嘴**，你独立走完：Plan Mode 探索计划 → 审阅纠偏 → 批准编码 → 审校员过一遍 → commit。走不下去再求助，每次求助记一笔（这就是你的薄弱点清单）
4. 上线：GitHub 建远程仓库、推送；愿意的话开 GitHub Pages
5. **c01 选做作业**（claude.ai 侧）：用 Research 调研「网文拆解方法论」并与你的 SOP 对照；用 Artifacts 给报告页做一版视觉原型；建一个 claude.ai Project 存放本项目资料
6. **终局三件套**：回 ai-learn 喊「点亮 p2」→ 写复盘笔记到 `practice/拆书工坊-复盘.md`（费曼输出：假想向一位工程师讲清整个系统怎么运转 + 五个阶段各自最卡的点）→ 教练按知识点覆盖矩阵逐项验收

**知识点回收清单**：
- [ ] MCP 的价值与上下文代价，三种作用范围（k0209）
- [ ] 二刷时哪些步骤不需要求助了（对照阶段 2）
- [ ] 能向别人完整讲清这个系统（毕业标准）

## 五、开工方式

每个阶段开一个新的 Claude Code 会话（本身就是上下文管理的实践），开场对教练说：**「拆书工坊，开始阶段 N」**——教练会读本方案，按当阶段教案带练。中途隔了几天也没关系，方案就是存档点。
