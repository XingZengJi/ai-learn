搭建 RAG 流水线时,你很快会发现,单靠语义搜索并不总能返回最好的结果。有时候你需要的是精确的关键词匹配,而这恰恰是语义搜索可能会漏掉的。解决办法是把语义搜索和词法搜索结合起来,用到的技术叫 BM25。

## 单靠语义搜索的问题

假设你要在一份文档里搜索一个具体的事件编号,比如「INC-2023-Q4-011」。语义搜索擅长理解上下文和含义,但它可能会返回一些语义上相关、却根本不包含你要找的确切词条的章节。

在这个例子里,语义搜索返回了网络安全那一节(确实包含这个事件编号),但也返回了一节完全没提到这起事件的财务分析内容。这是因为语义搜索关注的是概念上的相似性,而不是精确的词条匹配。

## 混合搜索策略

解决办法是同时运行语义搜索和词法搜索,再把两者的结果合并。这样你能同时拿到两种方法的优点:

- 语义搜索用嵌入向量找出概念上相关的内容
- 词法搜索用经典的文本搜索找出精确的词条匹配
- 合并结果把两种方法结合起来,得到更好的准确度

## BM25 是如何运作的

BM25(Best Match 25)是 RAG 系统里做词法搜索最常用的算法之一。它处理搜索查询的过程是这样的:

**第一步:对查询做分词(Tokenize)**
把用户的问题拆成一个个独立的词条。比如,「a INC-2023-Q4-011」会被拆成 `["a", "INC-2023-Q4-011"]`。

**第二步:统计词频**
看每个词条在你所有文档里出现的频率。像「a」这种常见词可能出现 5 次,而「INC-2023-Q4-011」这种特定词条可能只出现一次。

**第三步:按重要性给词条加权**
出现频率越低的词条,重要性得分越高。「a」因为太常见,重要性很低;「INC-2023-Q4-011」因为很罕见,重要性很高。

**第四步:找出最佳匹配**
返回那些包含更多「高权重词条」的文档。

## 实现 BM25 搜索

下面是搭建一个基础 BM25 搜索系统的方式:

```python
# 1. 按章节把文本分块
chunks = chunk_by_section(text)

# 2. 创建一个 BM25 存储,加入文档
store = BM25Index()
for chunk in chunks:
    store.add_document({"content": chunk})

# 3. 搜索
results = store.search("What happened with INC-2023-Q4-011?", 3)

# 打印结果
for doc, distance in results:
    print(distance, "\n", doc["content"][:200], "\n----\n")
```

运行这个搜索,你会得到比单纯语义搜索好得多的结果。BM25 算法会优先返回那些真正包含你specific 搜索词条的章节,尤其是像事件编号这类罕见词条。

注意,现在的结果正确地把「软件工程」那一节和「网络安全」那一节排在了前面——这两节都确实包含你要搜索的事件编号。

## 为什么这种方法效果更好

BM25 擅长找精确匹配,因为它:

- 给罕见、特定的词条更高的权重
- 忽略那些没有搜索价值的常见词
- 关注词频而不是语义含义
- 对技术术语、编号、特定短语尤其有效

核心洞察是:这两种搜索方法的优势是互补的。语义搜索理解上下文和含义,而词法搜索能确保你不会漏掉精确的词条匹配。把两者结合起来,你就能搭建出一个更健壮的搜索系统,既能处理概念性的查询,也能有效处理具体的精确查找。

下一步,你会学习如何合并这两套搜索系统的结果,打造统一的混合搜索体验。

---

对产品经理来说,BM25 和语义搜索的关系,类似电商搜索里「精确匹配商品型号」和「模糊理解购物意图」的区别:用户搜「iPhone 15 Pro 256G」时,你既想精确命中这个具体型号(词法搜索的强项),又想在用户搜「适合摄影的手机」时理解意图、推荐相关商品(语义搜索的强项)。只做一种都会漏掉另一类查询,这正是「混合搜索」被普遍采用的原因。
