检索增强生成(Retrieval Augmented Generation,简称 RAG)是一种技术,帮你处理那些太大、装不进单个提示词里的文档。RAG 不是把所有内容硬塞进一个超大的提示词,而是把文档拆分成一个个小块,回答问题时只挑出最相关的那几块塞进去。

## 大文档带来的问题

设想你有一份 800 页的财务文档,想问 Claude 一些具体问题,比如「这家公司有哪些风险因素?」。你需要想办法把文档里相关的信息传给 Claude,但一个提示词能装下的文本量是有上限的。

### 方案一:把全部内容塞进提示词

第一种做法很直接——把文档全部文字提取出来,连同用户的问题一起塞进提示词。提示词大概长这样:

```
Answer the user's question about the financial document.

<user_question>
{user_question}
</user_question>

<financial_document>
{financial_document}
</financial_document>
```

这种方法有几个严重的局限:

- 提示词长度有硬性上限——你的文档可能太长了塞不进去
- 提示词太长时,Claude 的表现会下降
- 更大的提示词处理成本更高
- 更大的提示词处理耗时更长

### 方案二:把文档拆分成小块

RAG 采用了更聪明的做法。首先,在预处理阶段把文档拆成一个个小块。然后,当用户提问时,找出和问题最相关的那几块,只把这些放进提示词里。

具体是这样运作的:如果有人问「这家公司面临哪些风险?」,你会在这些小块里搜索,找到「风险因素」那一节,只把这个相关小块放进提示词。

## RAG 的优势

- Claude 能专注在最相关的内容上
- 能扩展到处理非常大的文档
- 能同时处理多份文档
- 更小的提示词成本更低、跑得更快

## RAG 的挑战

- 需要一个预处理步骤来拆分文档
- 需要一套搜索机制来找出「相关」的小块
- 被选中的小块可能没能包含 Claude 需要的全部上下文
- 拆分文本的方式有很多种——到底哪种最好?

比如,你可以把文档按等长的片段拆分,也可以根据文档结构(比如标题和章节)来拆分。每种方式都有各自的取舍,需要针对你的具体场景去评估。

## 什么时候该用 RAG

RAG 涉及很多技术决策,要付出的工作量比「直接把所有内容塞进提示词」要大得多。你需要分析:对你的具体应用来说,这些好处是否值得付出这份复杂度。当你要处理非常大的文档、多份文档,或者需要在成本和性能上做优化时,RAG 会特别有价值。

核心洞察是:RAG 是用「更多的实现复杂度」去换「可扩展性和效率」。虽然要把它做对需要更多前期工作,但它能让你处理那些用「简单粗暴塞进提示词」根本不可能处理的文档集合。

---

对产品经理来说,RAG 的取舍就像做搜索功能时「全量返回」还是「先筛选再展示」的选择——用户问一个问题时,与其把整本知识库甩给客服(或 AI),不如先用搜索引擎定位到最相关的几段,再呈现给客服参考。文档小的时候不用这么麻烦,但文档一旦大到一定规模,不筛选就直接效率低下、成本失控。
