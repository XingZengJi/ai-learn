# Building with the Claude API - 50 Prompt caching in action 提示词缓存实战

> Course: Building with the Claude API · Lesson 50
> 课程: Building with the Claude API · 第 50 课
> 来源: Anthropic Academy (anthropic.skilljar.com),用户登录后提供英文原文,2026-07-17 整理翻译存档

提示词缓存是一项强大的优化功能,当你反复给 Claude 发送相同内容时,它能让 API 请求既更快、又更便宜。我们来看看如何在应用里有效地实现它。

## 提示词缓存的运作方式

开启提示词缓存后,第一次请求会把内容写入一个有效期为一小时的缓存。后续请求可以直接从这个缓存里读取,而不需要重新处理同样的内容。这在你发送以下内容时特别有价值:

- 大型系统提示词(比如一个 6K token 的编程助手提示词)
- 复杂的工具 schema(多个工具加起来大约 1.7K token)
- 重复出现的消息内容

核心洞察是:只有当你反复发送完全相同的内容时,缓存才有帮助——但在很多应用里,这种情况发生得非常频繁。

## 配置工具 Schema 缓存

要缓存你的工具 schema,你需要给工具列表里的最后一个工具加上缓存控制字段。下面是不改动原始工具定义的正确做法:

```python
if tools:
    tools_clone = tools.copy()
    last_tool = tools_clone[-1].copy()
    last_tool["cache_control"] = {"type": "ephemeral"}
    tools_clone[-1] = last_tool
    params["tools"] = tools_clone
```

这个做法在加缓存控制字段之前,先复制了工具列表和最后一个工具的 schema。虽然你也可以直接修改 `tools[-1]["cache_control"]`,但用复制的方式能避免以后重新排列工具顺序时出问题。

## 系统提示词缓存

对于系统提示词,你需要把它构造成一个带缓存控制的文本块:

```python
if system:
    params["system"] = [
        {
            "type": "text",
            "text": system,
            "cache_control": {"type": "ephemeral"}
        }
    ]
```

这样就把你的系统提示词,从一个简单的字符串,转换成了支持缓存的结构化格式。

## 理解缓存的行为表现

开启缓存后运行请求,你会在响应里看到不同的用量模式:

- **第一次请求**:`cache_creation_input_tokens=1772` —— Claude 正在写入缓存
- **后续请求**:`cache_read_input_tokens=1772` —— Claude 正在从缓存里读取
- **内容变化时**:会出现新的缓存创建 token

缓存对变化极其敏感——哪怕只改动工具或系统提示词里的一个字符,都会让该组件对应的整个缓存失效。

## 缓存的顺序和断点

你可以在一次请求里设置多个缓存断点,顺序如下:

1. 工具(如果提供了)
2. 系统提示词(如果提供了)
3. 消息

如果你改动了系统提示词,但保持工具不变,你会看到「部分缓存读取」(工具部分)加上「缓存写入」(新的系统提示词部分)。这种精细化的缓存方式意味着,你只需要为真正变化的部分付处理成本。

## 实践中的考量

提示词缓存在以下情况下效果最好:

- 请求之间工具 schema 保持一致
- 系统提示词保持稳定
- 应用会用相似的上下文发起多次请求

记住,缓存只保留一小时,所以它是为「API 调用相对频繁」的应用设计的,而不是用于长期存储。

---

对产品经理来说,「缓存对变化极其敏感」这一条值得反复强调给团队:哪怕只是给系统提示词加了一个空格、改了一个措辞,都会让整个缓存失效、当次请求成本和延迟都恢复到未缓存状态。这意味着「频繁调整系统提示词做 A/B 测试」和「靠缓存省成本」这两件事本身是有摩擦的,产品和工程团队在做提示词迭代节奏的决策时,需要把这个成本权衡考虑进去。
