链式工作流(chaining workflows)乍看之下可能显得理所当然,但它实际上是你在使用 Claude 时会遇到的最有用的模式之一。当你要处理复杂任务、或者 Claude 很难稳定处理的长提示词时,这种方式会特别有价值。

## 什么是工作流链式拆解?

一个链式工作流,会把一个庞大、复杂的任务,拆解成一连串更小、按顺序执行的子任务。你不是要求 Claude 一次性搞定所有事情,而是把工作拆成一个个聚焦的步骤,层层递进。

来看一个实际的例子:假设你在搭建一个能自动生成并发布视频的社交媒体营销工具。与其把所有事情塞进一个巨大的提示词里让 Claude 处理,你可以这样拆解:

1. 在 Twitter 上寻找相关的热门话题
2. 挑出最有意思的话题(用 Claude)
3. 研究这个话题(用 Claude)
4. 为一个短视频写脚本(用 Claude)
5. 用 AI 虚拟形象和文字转语音制作视频
6. 把视频发布到社交媒体

## 为什么要链式拆解,而不是用一个大提示词?

你可能会想,为什么不把所有 Claude 相关的任务合并成一个提示词?关键的好处在于「专注度」——当你一次只给 Claude 一个具体任务时,它能集中精力把这一件事做好,而不用同时兼顾多个要求。

链式方法有几个优势:

- 把大任务拆成更小、无法并行处理的子任务
- 可以在每个任务之间,可选地插入非 LLM 的处理步骤
- 让 Claude 始终专注在整体任务的某一个方面

## 长提示词的问题

这正是链式拆解真正体现价值的地方。你经常会遇到这样的情况:需要 Claude 写内容,并且带有很多具体的限制条件。假设你想让 Claude 写一篇技术文章,你要求它:

- 不要提到这是 AI 写的
- 不要用表情符号
- 避免陈词滥调或过于随意的语言
- 用专业、技术性的语气来写

即便这些限制条件都写得清清楚楚,Claude 仍然可能产出违反其中某些规则的内容。你可能收到一篇仍然用了表情符号、提到了 AI 作者身份、或者读起来不够专业的文章。

## 链式拆解的解决方案

与其在一个庞大的提示词里死磕,不如用一个两步的链式方法:

**第一步**:发送你最初的提示词,并接受第一次的结果可能不完美这个事实。Claude 会生成一篇文章,但它可能违反了你的某些限制条件。

**第二步**:发起一个专门用来修正问题的后续请求。把 Claude 刚写的文章提供给它,给出针对性的修改指令:

```
Revise the article provided below. Follow these steps to rewrite the article: 1. Identify any location where the text identifies the author as an AI and remove them 2. Find and remove all emojis 3. Locate any cringey writing and replace it with text that would be written by a technical writer
```

这个方法之所以有效,是因为 Claude 现在能把全部注意力放在「修改」这一件事上,而不用同时兼顾「内容创作」和「遵守限制条件」两件事。

## 什么时候该用链式拆解

链式工作流在以下场景特别有用:

- 你有一个带多项要求的复杂任务
- Claude 在长提示词里持续忽略某些限制条件
- 你需要在步骤之间处理或校验输出
- 你想让每一次交互都保持聚焦、易于管理

链式拆解看起来像是多做了一步工作,但它往往比试图把所有东西塞进一个提示词里,能产出更好的结果。关键在于识别出:一个任务是不是复杂到值得被拆解成聚焦的、按顺序执行的多个步骤。

---

对产品经理来说,链式工作流解决的是一个很常见的管理问题:让一个人同时兼顾「创作」和「合规审查」两件事,质量往往两头都打折扣;更好的做法是让 ta 先专心创作,再单独安排一轮「按检查清单逐条修改」的审查环节——这和内容团队里「先写初稿,再由专人过一遍合规和风格检查」的分工逻辑完全一致。当你发现 AI 反复漏掉提示词里某条要求时,别急着把提示词写得更长更细,先想想能不能拆成两步来做。
