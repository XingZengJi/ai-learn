工作流(workflows)和智能体(agents),是处理「Claude 没法一次请求就搞定」的用户任务的两种策略。其实在整个课程中,你已经不知不觉地做过这两种东西了——当你使用工具、放手让 Claude 自己想办法完成任务时,那就是一个 agent。

## 什么时候用工作流,什么时候用智能体

这个决定,取决于你对这个任务的理解有多透彻:

- 当你能清楚地想象出 Claude 解决问题应该经过的确切流程或步骤时,用**工作流**;或者当你的应用 UX 本身就把用户限定在一组固定任务里时,也适合用工作流
- 当你不确定自己到底会给 Claude 什么样的任务或任务参数时,用**智能体**

工作流是一系列对 Claude 的调用,通过预先设定好的一连串步骤来解决一个具体问题。智能体则是给 Claude 一个目标和一套工具,期望 Claude 自己想办法,通过这些工具去完成目标。

## 例子:图片转 CAD 的工作流

来看一个实际的工作流例子。想象你在搭建一个网页应用,用户拖拽上传一张金属零件的图片,你要据此生成一个 STEP 文件(3D 模型的行业标准格式)。

由于我们对「用户提供了一张图片后该做什么」有相当清晰的把握,而且可以很容易地把整个过程写成代码里预先设定好的一系列步骤,这就是一个完美的工作流候选场景。

工作流的拆解如下:

1. 把图片喂给 Claude,让它描述这个物体
2. 基于这份描述,让 Claude 用 CadQuery 库对这个物体建模
3. 生成渲染图
4. 让 Claude 对照原始图片给渲染图打分。如果有问题,就修正它

## 评估-优化模式(Evaluator-Optimizer Pattern)

这个建模工作流,正是「评估-优化模式」的一个例子。运作方式如下:

- **生产者(Producer)**:接收输入、产出输出(Claude 用 CadQuery 给零件建模、生成渲染图)
- **评估者(Grader)**:按某种标准评估这份输出
- **反馈循环(Feedback loop)**:如果评估者不认可这份输出,反馈会传回给生产者,要求改进
- **迭代(Iteration)**:这个循环持续进行,直到评估者认可输出为止

## 为什么要学习工作流模式

识别不同工作流模式的目的,是给你一套可复用的「配方」,方便你自己实现功能时套用。「评估-优化」正是一种已经在其他工程师那里验证有效的工作流模式——不妨考虑在你自己的应用里用一用!

记住,识别出工作流本身并不会自动帮你做成什么——你仍然需要写实际的代码去实现它。但这些模式已经在很多工程师身上被证明是成功的,所以值得理解、并应用到你自己的项目里。

---

对产品经理来说,「工作流 vs 智能体」的选择,本质上是「能不能提前把 SOP 写清楚」的判断:如果一件事的处理步骤是固定、可预见的(比如「客户退款申请」的标准流程),就该做成工作流——每一步都受控、可预测、易调试;如果任务边界模糊、每次的需求都可能不一样(比如「帮我研究一下这个新市场」),就该用智能体模式,给它目标和工具,让它自己判断怎么走。把该走流程的事硬做成智能体、或把该灵活处理的事硬套死流程,都会导致产品体验打折扣。
