## 课程结构

**这门课的主线是一根「你自己做多少、平台替你做多少」的轴**,从头到尾单向推进:

1. **打底**(01–03)—— 平台三层结构、一次最小 API 调用、按成本选模型
2. **教会你的智能体**(04–06)—— **手写智能体循环** → 工具使用(含 tool runner) → 扩展思考
3. **扩展你的智能体**(07–10)—— 内置工具(Anthropic 执行) → Skills(教流程) → MCP(接第三方) → 上下文管理(四种模式)
4. **托管智能体**(11–12)—— **整个循环交给 Anthropic 跑**
5. **收尾**(13)—— 干脆让 Claude Code 把代码也写了

**这根轴的三个刻度值得单独记**: 第 4 课手写 `while` 循环 → 第 5 课 tool runner 接管循环 → 第 12 课托管智能体接管整个智能体。**同一件事,委派程度递增。**

## 几条值得单独记住的

**build with primitives, scale on infrastructure, run with control**(01)—— 平台三层的记法。**大多数团队只在原语层做决策就开工,然后在基础设施层撞墙。**

**响应的 content 是块数组,不是字符串**(02)—— 因为一次回复里可能同时有思考、工具调用、文本。**智能体循环本质就是在这个数组里找特定类型的块并做出反应。**

**正确的模型 = 输出你真的愿意发出去的那些里最便宜的一个**(03)—— 配套做法: 从真实负载取 **20–30 个例子**做评估,**从 Haiku 往上跑,达标即停**。生产环境里**在同一个接口内按任务路由到不同模型**。

**`response.usage` 让成本可观测**(03)—— 上线第一天就能知道单次调用的实际 token 消耗,不用等月底账单。

**你拥有循环和工具,Claude 拥有推理**(04)—— 划清了责任边界,也解释了为什么智能体可靠性主要取决于你的代码。

**工具描述含糊是智能体失灵的头号原因**(05)—— 把一类看似模型能力的问题,重新定位成了**文档写作问题**。这条在第 8 课(Skill 的描述)又出现一次。

**tools 关乎能做什么,Skills 关乎希望怎么做**(08)—— 再加上第 9 课: **tools 管你自己的东西,skills 管你的流程,MCP 管别人的东西。**

**MCP 的价值是维护责任的归属**(09)—— 不是功能差异,而是**三年后谁在改代码**。配套技巧: **`default_config: {"enabled": False}` 默认全关再逐个开**,一个配置同时解决权限与上下文占用两个问题。

**上下文管理的四种模式各对应一个失败模式**(10)—— **成本 → 提示词缓存;窗口大小 → 压缩 / 即时上下文;无状态 → 记忆工具。** 先确定坏的是哪个再选手段。

**有取舍的用扩展思考,没取舍的不用**(06)—— 分类抽取只有一个正确答案,思考帮不上忙;**「这两条规格是否冲突」才是它挣回成本的地方。**

**你定义什么算完成,Claude 干到那为止**(11)—— rubric + 独立 grader + 自动迭代。**注意评分器跑在自己的上下文窗口里**,不受被评估内容影响。这与 `Course/20AIforB/` 的「先写验收测试再写代码」是同一件事。

**先开事件流,再发起始消息**(12)—— 流只投递打开之后的事件。**典型的「本地碰巧没事、线上偶发丢事件」型的坑。**

**好提示词点名三件事: 文件、模式、最终状态**(13)——「描述要具体」在本课的第三次出现。
