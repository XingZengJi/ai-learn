# Claude Platform 101 - 11 What are managed agents? 什么是托管智能体

> Course: Claude Platform 101 · Lesson 11 · 章节: Managed Agents
> 课程: Claude Platform 101(Claude 开发者平台入门)· 第 11 课
> 来源: Anthropic Academy (anthropic.skilljar.com),登录后抓取英文原文整理翻译,2026-07-24

**Claude Managed Agents 是一套用于大规模构建与部署智能体的 API。** 你定义带特定工具、人设与能力的智能体,配置装好所需包与网络控制的沙箱环境,**然后从你自己的应用里发起会话**,Claude 在一个隔离容器里干活——容器里有完整的文件系统访问、bash 执行与网络搜索。

## 那个智能体循环,替你托管了

底层还是**智能体循环**: Claude 推理、调工具、读结果、重复,直到任务完成。**如果你之前手写过智能体,这个循环你写过。托管智能体把同样的循环放到 Anthropic 的基础设施上跑,你不用自己运行它。**

在 Claude Console 里,Managed Agents 有独立的一块。

理解它能解锁什么,最好的方式是看几个例子。

## 例子 1: 一块会自己干活的看板

![看板示例](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2F8lsy243ftffjjy1cx9lm3o2bw%2Fpublic%2F1780966787%2F10b-what-are-managed-agents_01.1780966787467.png)

设想一块建在托管智能体之上的看板。**你把一张票拖进「进行中」,就自动触发一个会话。** 假设票上写着「优化网站性能」,接下来:

- 你的后端**创建一个 session**
- 会话指向一个**你配置好的环境**,里面预装了 Lighthouse 和 Puppeteer
- **你的 GitHub 仓库被挂载进容器**

现在 Claude 有了代码库、工具,以及一份**定义「什么算完成」的评分标准(rubric)**:

- Lighthouse 分数高于 90
- 没有阻塞渲染的资源
- 所有图片懒加载

Claude 跑完审计,然后开始压缩图片、内联 CSS、延迟脚本。**每一次工具调用都通过事件流实时回传到看板上,你能看着它干。**

**接着评分标准生效**: 一个**独立的评分器(grader),跑在自己的上下文窗口里**,按你的标准评估输出。Claude 读到反馈、回去补上遗漏的、再次提交。**演示里这个循环把 Lighthouse 分数推到了 96。**

还有一点: **第一张票还在跑的时候,你可以再拖第二张进来。两个会话、两个容器、两个任务并行。**

## 例子 2: 带记忆的周期性调研智能体

![调研智能体](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2F8lsy243ftffjjy1cx9lm3o2bw%2Fpublic%2F1780966788%2F10b-what-are-managed-agents_08.1780966788017.png)

另一种形态的智能体: 追踪公司付费的每个 SaaS 工具的**价格与套餐变动**,在站会之前把报告准备好。

每次运行时,这个智能体会:

- **搜索网络**上的当前定价页、检查套餐层级变化、标记可能影响合同的新功能
- **在沙箱里用 Python 跑成本分析**
- **用一个 Excel 表格 skill** 并撰写执行摘要
- **通过 MCP 服务器**把链接发到 Slack、在 Asana 建一个复核任务

![记忆](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2F8lsy243ftffjjy1cx9lm3o2bw%2Fpublic%2F1780966788%2F10b-what-are-managed-agents_10.1780966788532.png)

智能体还会**读写一个记忆存储**: 开工前先看上周发现了什么,结束后存下变化了什么。**于是下周一的报告能说「算力成本比上周低了 15%」,而不是把同样的静态价格再列一遍。**

## 例子 3: 多智能体协同的事故响应

![多智能体](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2F8lsy243ftffjjy1cx9lm3o2bw%2Fpublic%2F1780966789%2F10b-what-are-managed-agents_11.1780966788976.png)

监控系统触发告警,后端的一个自定义工具接到告警载荷,**把它作为工具结果送进一个新会话**。这个会话用到**多智能体协同**:

- 一个**协调者(coordinator)** 接到告警,分派给三个专家
- **每个专家跑在自己的上下文窗口里,共享同一个文件系统**
- 专家们汇报回来,**协调者把各方发现综合成一份事故摘要**

**摘要发到 Slack 之前,权限策略生效**: 你在屏幕上看到草稿、批准,消息才发出去。**敏感操作会等人。**

**记忆把这一切串起来**: 协调者查阅记忆里过往的事故,标记出一个模式——「这看起来像两周前那次 DNS 解析问题,起因是 TTL 配置错误」。**下次类似告警触发时,智能体带着这个上下文起步,而不是从零诊断。**

## 构成模块

跨越这几个例子,托管智能体提供的是一套完整的、有状态的托管智能体体验,建立在:

| 模块 | 是什么 |
|---|---|
| **Agents** | 带特定工具、人设与能力的定义 |
| **Sessions** | 你从自己应用里发起的单次运行 |
| **Environments** | 装好所需包与网络控制的沙箱 |
| **Tools** | 包括你后端的自定义工具 |
| **MCP** | 连到 Slack、Asana 这类服务 |
| **Memory** | 智能体开工前读、完工后写的存储 |
| **Outcomes** | **定义并检查「什么算完成」的评分标准与评分器** |
| **Multi-agent coordination** | 协调者分派给专家 |

## Recap 本课要点

- **托管智能体是一套用于大规模构建与部署智能体的 API,托管在 Anthropic 的基础设施上**
- 它跑的还是那个熟悉的智能体循环——**推理、调工具、读结果、重复**——只是在一个带文件系统、bash 执行与网络搜索的**隔离容器**里
- **会话运行在你配置的环境中,可以并行,并把工具调用实时流回你的应用**
- **评分标准与独立评分器让你定义成功条件,Claude 会反复迭代直到达标**
- 记忆、MCP、自定义工具、权限策略、多智能体协同,共同构成有状态的体验
- **你定义什么算完成,Claude 干到那个程度为止**

## 对产品经理来说

**「你定义什么算完成,Claude 干到那为止」——这是本课的核心转变,也是对 PM 影响最大的一句。**

前十课的模式是「**你发指令,它给产出,你评估**」。托管智能体把评估这一环也自动化了: **rubric(评分标准)+ 独立 grader(评分器)+ 自动迭代。** 注意例子里那个细节——**评分器跑在自己的上下文窗口里**,也就是说它不受被评估内容的影响,是个独立的判断者。

**这直接改变了需求的写法。** 以前你写「优化网站性能」,然后人工验收;现在你写的是**可判定的完成标准**(Lighthouse > 90、无阻塞渲染资源、图片全部懒加载)——**这和 `Course/20AIforB/` 第 4 课「先写验收测试再写代码」是同一件事,只不过在这里,验收测试成了驱动智能体迭代的燃料。**

**两个值得注意的产品设计点:**

- **「敏感操作会等人」**(权限策略)—— 事故响应那个例子里,摘要发出前要人批准。**这是把「人在回路」做成了基础设施能力,而不是靠自觉**
- **记忆让智能体产出从「快照」变成「趋势」** —— 「算力成本比上周低 15%」和「当前价格是 X」是完全不同量级的产出价值,**而差别只是有没有记忆存储**

**什么时候不用它?** 下一课会讲清楚这条边界。简单说: 循环短、要完全控制时,自己写。
