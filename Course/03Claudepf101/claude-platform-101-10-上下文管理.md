# Claude Platform 101 - 10 Context management 上下文管理

> 来源: Anthropic Academy (anthropic.skilljar.com),登录后抓取英文原文整理翻译,2026-07-24

每个请求都有**上下文窗口**。**一百万 token 听起来很多,但一旦你真的在跑一个智能体,它耗尽的速度比你想的快。** 上下文管理就是**在不丢掉要紧东西的前提下待在窗口内**的办法。

## 什么算作上下文

![上下文构成](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2F8lsy243ftffjjy1cx9lm3o2bw%2Fpublic%2F1780966785%2F10-context-management_03.1780966785391.png)

**上下文是 Claude 在某一轮里看到的一切:**

- 系统提示词
- 消息历史
- **工具定义与工具结果**
- 附加的文件与 skills
- **思考块**

**它是每一次 API 调用的输入。进去要付费,出来也要付费。而一旦窗口满了,请求就失败。**

> **所以目标不是把所有东西塞进去,而是把对的东西塞进去。**

![四种模式](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2F8lsy243ftffjjy1cx9lm3o2bw%2Fpublic%2F1780966786%2F10-context-management_05.1780966785982.png)

Anthropic 公布了四种用于长时运行智能体的上下文管理模式。**三个是一等 API 特性,一个是设计模式。**

## 模式 1: Just-in-time context 即时上下文

**别一开始就把所有东西载入。载入智能体当下需要的,其余让它通过工具在需要时自己拉。**

以合规审查智能体为例: **它不会把整本建筑规范塞进系统提示词**,而是在需要某一条时调用 `lookup_building_code` 工具。

**这是四种里唯一的「设计模式」**——API 里没有对应特性,**纯粹是你对「载入什么、何时载入」的刻意选择。**

## 模式 2: Server-side compaction 服务端压缩

对话变长时,**Anthropic 的服务端压缩会把旧的轮次总结成一个块。** 在请求里加 `context_management` 键来启用:

```python
response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    context_management={
        "edits": [
            {"type": "compact"}
        ]
    },
    messages=messages,
)
```

**输入越过触发阈值时,API 自动做总结。你不必自己跟踪对话长度。**

## 模式 3: Prompt caching 提示词缓存

**缓存让你标记出请求里稳定的部分**——系统提示词、工具定义、一份长文档——**然后在多次调用间以极低的成本复用它们。**

> **这里的算术比看上去重要**: 如果系统提示词是 4,000 token、每小时调用 100 次,**缓存就是「账单可接受」和「财务打电话过来」之间的差别。**

## 模式 4: The memory tool 记忆工具

**有些上下文需要跨会话存活**: 用户偏好、智能体的工作笔记、上周定下的事。推荐的原语是**记忆工具**。

![记忆工具](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2F8lsy243ftffjjy1cx9lm3o2bw%2Fpublic%2F1780966786%2F10-context-management_15.1780966786657.png)

它怎么运作:

- **Claude 通过工具调用读写一个记忆目录**
- **存储后端由你在客户端实现**——文件系统、数据库、加密存储,随你
- **Anthropic 会自动注入一条系统指令**,告诉 Claude 开工前先检查记忆目录

## 把模式叠起来用

真实应用里**通常四种一起上**。那个合规审查智能体会**缓存系统提示词与工具定义**,并**通过 `lookup_building_code` 即时拉取规范条款**。

> **每种模式对应一个不同的失败模式: 成本、窗口大小、无状态。挑与你正在坏掉的那个匹配的。**

## Recap 本课要点

- **上下文是 Claude 在一轮里看到的一切,既不免费也不无限。窗口满了请求就失败**
- **即时上下文**: 载入当下需要的,其余让工具去拉。**四种里唯一的设计模式**
- **服务端压缩**: 加 `context_management` 键,输入越过阈值时 API 自动总结旧轮次
- **提示词缓存**: 标记请求里稳定的部分,以极低成本跨调用复用
- **记忆工具**: Claude 通过工具调用读写记忆目录,**存储后端归你**,上下文得以跨会话存活
- **四种模式,一个目标。可以手工接,也可以用托管智能体——它默认就开着缓存和压缩**

## 对产品经理来说

**「每种模式对应一个不同的失败模式: 成本、窗口大小、无状态」——这句是本课最好的诊断工具。** 上下文管理听起来像一个笼统的技术话题,但它其实是三个独立问题:

| 症状 | 真正的问题 | 对应手段 |
|---|---|---|
| **账单太高** | 成本 | **提示词缓存** |
| **长对话跑着跑着报错** | 窗口大小 | **服务端压缩、即时上下文** |
| **每次都要用户重新交代一遍** | 无状态 | **记忆工具** |

**先确定坏的是哪一个,再选手段**——而不是把四个都上。

**缓存那笔账值得 PM 自己算一遍。** 4,000 token 的系统提示词 × 每小时 100 次调用,这个量级在任何有点流量的功能里都很普通,**而缓存与否的差别是数量级的。** 这属于「上线前就该确认、上线后才发现会很痛」的那类配置。

**「即时上下文是设计模式而非 API 特性」这个提示也很关键**: 它意味着这一条**没有开关可以打开**,只能靠架构上的克制。**倾向于「把所有可能用到的资料都塞进系统提示词」是最常见的反模式**,它在 demo 阶段完全看不出问题,规模化后同时引爆成本和窗口两个问题。
