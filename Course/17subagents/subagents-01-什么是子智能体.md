## 定义

子智能体是 Claude Code 可以把任务**委派**出去的专门助手。可以把它们理解成**专注的帮手**: 每一个都在**自己独立的对话上下文窗口**里运行,干完活,把一份摘要交回主线程。

中间过程——所有的文件读取、搜索、工具调用——**全部被隔离,永远不会弄乱你的主对话**。

## 为什么这件事重要

每次你和 Claude Code 对话,都在往主上下文窗口里堆东西。每一次工具调用、每一次文件读取、每一个搜索结果都存在那儿。**那个空间是有限的**,一旦填满,Claude 就开始丢失对话早期的内容。

子智能体的解法是**另开一个上下文窗口**。子智能体收到两样东西:

1. **来自你配置文件的自定义系统提示词** —— 定义它的角色与行为
2. **父智能体根据你的请求写出的任务描述**

然后它自己干活: 读文件、跑搜索、改代码,需要什么做什么。完成后,**只有一份摘要回到你的主对话,整个子智能体对话随即被丢弃**。

这意味着你的主上下文保持干净——**你拿到了答案,却不必背上寻找答案一路上的噪音**。

**代价是**: 你失去了对「它是怎么得出结论的」的可见性。

## 一个实际例子

假设你在摸索一个不熟悉的代码库,想知道哪个服务负责处理退款。

- **不用子智能体**: Claude 可能读 15 个文件、跑好几次搜索、追踪多个函数调用。**这一切都填进你的上下文窗口**,尽管你只需要一个事实
- **用子智能体**: 你提问,Explore 子智能体启动,在自己的上下文里完成所有挖掘,交回一个聚焦的答案。**你的主上下文只记录了问题和摘要**,而不是沿途读过的那 15 个文件

## 内置子智能体

Claude Code 自带几个开箱即用的子智能体:

| 名称 | 用途 |
|---|---|
| **General purpose** | 需要同时探索和动手的多步任务 |
| **Explore** | 快速搜索与导航代码库 |
| **Plan** | 计划模式下,在给出方案前研究和分析代码库 |

## 自定义子智能体

除了内置的,你也可以创建自己的——带自定义系统提示词和工具权限。这让你能定义贴合自己工作流的专门智能体: 代码评审员、测试编写者、文档生成器,或任何你需要的角色。

## Key Takeaways 核心要点

子智能体带来三个主要好处:

- **把工作拆成聚焦的小块**,让每个子智能体专注于一个具体任务
- **保持主上下文窗口干净**,把所有中间工作隔离出去
- **只带回你需要的信息**,以简洁摘要的形式

无论用内置的还是自建的,它们都是让长时间 Claude Code 会话更高效的实用手段。**主上下文里的噪音越少,你能工作的时间越长、效率越高。**

## 对产品经理来说

理解子智能体,关键是理解**上下文窗口是一种有限资源**。这一课其实在讲一个资源管理问题: 你的主对话像一张会议桌,每读一个文件就往桌上堆一叠纸。桌子堆满了,你就开始找不到早先放的东西。子智能体的作用是**让助理去另一个房间翻资料,只把结论那一页拿回来**。

那个「15 个文件 vs 1 个答案」的例子很能说明性价比。**你付出的成本是可见性,换回的是空间。**

这个取舍在管理上有完全对应的版本: 把任务委派出去,你得到的是自己的精力和注意力,失去的是对过程的掌握。**什么时候该委派、什么时候该自己盯着**——判断依据是一样的:**过程本身重不重要**。这正是第 4 课要展开的决策规则。
