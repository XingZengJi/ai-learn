# Introduction to Subagents - 04 Using subagents effectively 何时该用、何时别用

> Course: Introduction to Subagents(Subagents 入门)· Lesson 4
> 课程: Introduction to Subagents · 第 4 课
> 来源: Anthropic Academy (anthropic.skilljar.com),登录后抓取英文原文翻译整理,2026-07-23

会创建、会设计之后,真正的问题是: **它们什么时候真的帮上忙,什么时候反而添乱?** 课程给的答案归结为一件事——**中间过程对你的主线程重不重要。**

## 什么时候子智能体最出彩

子智能体最擅长的是**探索与执行相互分离**的场景。

**如果任务的每一步都依赖上一步的发现,那这些工作就该留在主线程。但如果你只要一个答案、不在乎过程,就委派出去。**

适合的任务特征:

- 你要的是**结果**,不是「怎么找到的」的实况转播
- 探索过程会把主线程的上下文塞满
- 这个任务能从**全新视角**或**自定义系统提示词**中获益

## 场景一: 研究任务

研究是子智能体的经典用例。比如调查一个陌生代码库里认证是怎么实现的——**你的主线程需要知道 JWT 在哪里被校验,但不需要看到沿途搜索过的每一个文件。**

研究子智能体可以读几十个文件、追踪函数调用、探索不同代码路径,**所有这些探索都留在它自己的上下文里**。你的主线程收到的是一份干净的摘要,类似:

```
JWT validation happens in middleware/auth.js line 42,
called from the Express router in route/api.js
```

重活它干了,你的主线程拿到的正好是继续推进所需要的东西。

## 场景二: 代码评审

这一条背后有个很有意思的观察:

> **当代码被呈现为「别人写的」时,Claude 评审得更有效。**

如果你和主线程来回很多轮建起了一个功能,再让同一个线程去评审它,**得到的反馈往往很弱**——因为 Claude 参与了创造过程,很难用全新的眼光去看它。

评审子智能体在**独立上下文**里看这些改动: 跑 `git diff`、读被改的文件、套用它专门的评审标准,**不带任何关于这些代码是怎么被写出来的历史**。

这种分离还有个附加好处: 你可以把**项目专属的评审标准编码进子智能体的系统提示词**,确保全团队的评审标准一致。

## 场景三: 需要自定义系统提示词的任务

Claude Code 默认的系统提示词强调**简洁、聚焦代码**的回应。这对写代码很好,但不是对所有事都好。

两个自定义系统提示词让子智能体真正强过主线程的例子:

- **文案子智能体** —— 给它关于语气、受众、风格的指令。Claude Code 的默认提示词倾向于简洁的技术写作,**这真不是你写落地页或邮件营销想要的东西**。文案子智能体可以有一套完全不同的关于声音和结构的指令
- **样式子智能体** —— 让它指向你的设计系统文件。子智能体启动时这些文件自动载入它的上下文,**所以它在写第一行 CSS 之前就已经知道你的颜色变量、间距约定和组件模式**

## 什么时候子智能体反而有害

启动一个子智能体是有开销的——**失去对过程的可见性、把发现压缩成一份摘要**。只有当它能做到主线程做不到的事时,这个开销才划算。

课程点名了三种反模式:

### 反模式一: 声称专家身份

**声称专业身份的子智能体很少有用。** 类似「你是一个 Python 专家」「你是一个 Kubernetes 专家」这样的提示词**没有增加任何价值**,因为 Claude 本来就有这些知识。

**所谓的"专家"子智能体能做的事,你的主线程直接就能做。**

### 反模式二: 顺序流水线

顺序式的子智能体流水线会带来麻烦。设想一个三智能体流程: 一个复现 bug、一个调试、一个修复。

**流水线在任务真正独立时才有效。当每一步都依赖上一步的发现时就会失败——而修 bug 几乎总是这样。信息在智能体之间的交接中丢失。**

### 反模式三: 测试运行器

测试运行器子智能体倾向于**隐藏你需要的信息**。测试失败时,你想要完整的输出来诊断问题。一个只返回「测试失败了」的子智能体,**逼得你再去写额外的调试脚本**,才能拿到本来直接就能看到的细节。

课程明确指出: **在所有配置中,测试运行器这个模式的表现是最差的。**

## 决策规则

判断要不要用子智能体,问自己**一个问题**:

> ### **中间过程重要吗?**

- **不重要**(你只要最终结果)→ **委派给子智能体**
- **重要**(你需要看到并对过程中发生的事做出反应)→ **留在主线程**

| ✅ 该用 | ❌ 别用 |
|---|---|
| 研究与探索 | 不增加真实能力的「专家」人设 |
| 代码评审 | 每步依赖上一步的多步流水线 |
| 需要自定义系统提示词的任务 | 需要完整输出来调试的跑测试 |

## 对产品经理来说

这一课最有价值的地方在于它**明确说了什么时候不该用**——这在技术课程里很少见,大多数课只教你怎么用得更好。

那条决策规则「**中间过程重要吗**」干净到可以直接用,而且它在管理上有完全对应的版本: **你委派一件事时,你要的是结论还是要跟进过程?** 要结论就放手,要跟进就别委派——最糟的情况是**名义上委派了、实际上又不断打断去看进度**,那等于两头的成本都付了。

三个反模式里,**「专家人设」这条尤其值得记住**,因为它戳破了一个很流行的做法。给 AI 套一个"你是资深某某专家"的身份,感觉上像是在提升能力,**实际上什么也没增加**——模型的知识不会因为你这么说而变多。真正有效的自定义系统提示词是那两个例子里的东西:**给它默认不具备的上下文**(设计系统文件)或**改变它默认的行为倾向**(从技术简洁改成文案风格)。**身份标签不是能力,上下文和约束才是。**

「顺序流水线在交接中丢信息」也很有普遍性。**每一次交接都是一次信息损耗**,所以流水线只在环节真正独立时才划算。这跟第 3 课的「障碍汇报」是一体两面——那一课教的正是**怎么减少交接损耗**。
