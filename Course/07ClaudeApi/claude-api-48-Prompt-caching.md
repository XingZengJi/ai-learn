# Building with the Claude API - 48 Prompt caching 提示词缓存

提示词缓存(Prompt caching)是一项功能,通过复用之前请求的计算工作,来加快 Claude 的响应速度、降低文本生成的成本。它不是在每次请求后就把所有处理工作扔掉,而是当你再次发送类似内容时,让 Claude 能保存并复用这些工作。

## Claude 通常是如何处理请求的

要理解提示词缓存,我们先看看在没开启缓存的普通请求里,背后发生了什么。

当你给 Claude 发一条消息时,它并不会立刻开始生成回答,而是先对你的输入做大量的预处理工作:

1. 把提示词拆分成更小的片段(分词)
2. 为每个 token 生成嵌入向量
3. 根据上下文添加语境信息
4. 到这一步才真正开始生成输出文本

把回答发给你之后,Claude 会把这些计算工作全部扔掉——分词、嵌入、上下文分析,统统被丢弃。

## 「扔掉工作」带来的问题

当你发起包含相同内容的后续请求时,这种做法就变得低效了。比如,在一段对话里,你反复让 Claude 修改同一段长文本的摘要:

Claude 不得不对刚刚分析过的内容,重复一遍一模一样的预处理工作。仿佛 Claude 会想:「我刚处理过这条消息,却把做的工作全扔了——我本可以直接复用的!」

## 提示词缓存是如何解决这个问题的

提示词缓存改变了这套流程:把预处理的工作保存下来,而不是丢弃。

当你发起最初的请求时,Claude 照常完成所有预处理工作,但会把结果存进缓存,而不是扔掉。这个缓存就像一张查找表,相当于说:「如果我以后再看到这条消息,我就直接复用已经做好的工作。」

## 关键优势和限制

提示词缓存带来几个好处:

- **响应更快**:使用缓存内容的请求执行得更快
- **成本更低**:请求里被缓存的部分,你付的钱更少
- **自动优化**:最初的请求写入缓存,后续请求从缓存里读取

不过,也有几个重要的限制需要记住:

- **缓存有效期**:缓存内容只保留一小时
- **适用场景有限**:只有在你反复发送相同内容时才有意义
- **需要较高的重复频率**:当同样的内容在你的请求里出现得非常频繁时,效果最好

提示词缓存最适合的场景,比如「文档分析工作流」——你要针对同一份大文档反复提问;或者「迭代式编辑任务」——基础内容保持不变,你只是在反复打磨某些具体方面。

---

对产品经理来说,提示词缓存就像餐厅后厨的「半成品预制」——如果每天都要做同一道菜的基础处理(比如熬同一锅高汤),与其每次点单都从头熬一遍,不如提前做好、一小时内反复用。但这套机制只在「同样的内容反复被请求」时才划算——如果每次点的菜都完全不一样,预制半成品反而没有意义。评估要不要上这个优化,先看你的产品里有没有「高频重复的固定上下文」(比如系统提示词、工具定义)。
