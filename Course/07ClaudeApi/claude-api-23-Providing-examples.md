在提示词里提供示例,是你会用到的最有效的提示词工程技巧之一。这种方法被称为「单样本(one-shot)」或「多样本(multi-shot)」提示,做法是给 Claude 一些「输入-输出」的示例对,来引导它的回答方式。

## 示例是如何起作用的

来看一个情感分析的例子。假设你想让 Claude 判断一条推文是正面还是负面情绪。

这里的难点在于讽刺(sarcasm)。像「是啊,可不是嘛,这是我看过最好的电影了,上一次这种感觉还是看《外太空计划九号》的时候」这样一条推文,表面上看是正面的,实际上是讽刺、是负面情绪(《外太空计划九号》是公认的史上最烂电影之一)。

## 加示例来处理边界情况

要解决这个问题,你可以加入一些示例,展示 Claude 该如何处理这类刁钻的情况:

改进后的提示词包括:

- 一个明确的正面示例:「今晚这场比赛真精彩!」→「正面」
- 一个讽刺的示例:「哦耶,我今晚可太需要一次航班延误了!太棒了!」→「负面」
- 解释为什么讽刺需要特别对待的上下文说明

注意示例是包在 `<sample_input>` 和 `<ideal_output>` 这样的 XML 标签里的。这种结构让 Claude 能非常清楚地知道每部分分别代表什么。

## 什么时候该用示例

示例在以下场景特别有用:

- 捕捉边界情况或特殊场景
- 定义复杂的输出格式(比如特定的 JSON 结构)
- 展示你想要的确切风格或语气
- 演示如何处理模糊不清的输入

## 单样本 vs 多样本

- **单样本(One-Shot)**:提供单个示例来确立模式
- **多样本(Multi-Shot)**:提供多个示例来覆盖不同场景

当你需要处理各种边界情况、或想展示不同类型的有效回答时,用多样本。

## 从评估结果中找好示例

跑提示词评估时,可以留意那些得分最高的输出:

找到打了 10 分(或你评分体系里的最高分)的回答,把对应的输入-输出对拿来当作提示词里的示例。这能帮助 Claude 理解:对你这个具体场景来说,「完美」的输出长什么样。

## 给示例加上说明

不要只给出输入-输出对,还要解释为什么这个输出是好的:

```
<ideal_output>
[Your example output here]
</ideal_output>

This example is well-structured, provides detailed information 
on food choices and quantities, and aligns with the athlete's 
goals and restrictions.
```

这段额外的说明能帮 Claude 理解「好回答」背后的原因,而不只是格式本身。

## 最佳实践

- 始终用 XML 标签清晰地组织示例
- 明确说明你在展示什么:「这是一个示例输入,以及对应的理想回答」
- 加入能覆盖你最常见失败场景的示例
- 解释为什么示例输出被认为是理想的
- 保持示例与具体任务紧密相关

示例之所以特别有效,是因为它们「做给你看」而不是「说给你听」。与其试图用语言精确描述你想要什么,不如直接演示出来。这能让你的提示词更可靠,也能帮 Claude 理解那些用纯文字指令很难说清楚的细微要求。

---

对产品经理来说,「给示例」就是需求文档里的「验收案例(acceptance examples)」——比起在需求描述里堆砌形容词,不如直接给出「输入是这个,期望输出长这样」的具体案例,尤其是那些容易被误判的边界场景(比如讽刺、反话)。一个好案例胜过十句抽象描述。
