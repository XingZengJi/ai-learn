我们要搭建一个基于命令行(CLI)的聊天机器人,借此更好地理解 MCP 客户端和服务器是如何协同工作的。这个动手项目会让你同时获得 MCP 架构两端(客户端和服务器)的实践经验。

## 我们要搭建什么

我们的聊天机器人,会让用户通过命令行界面和一批文档进行交互。这套系统由两个主要部分组成:

- 一个 MCP 客户端,负责处理用户交互
- 一个自定义的 MCP 服务器,负责管理文档操作

服务器会提供两个核心工具:一个用来读取文档内容,另一个用来更新文档内容。为了简单起见,所有文档都存在内存里——不需要数据库。

## 重要的架构说明

在真实项目里,你通常只会实现「MCP 客户端」或「MCP 服务器」中的一个,而不是两个都做。你可能会:

- 做一个 MCP 服务器,把你的服务开放给其他开发者使用
- 做一个 MCP 客户端,去连接现有的 MCP 服务器

这个项目里我们两个部分都做,纯粹是出于教学目的——为了理解它们是如何互相通信、协同工作的。

## 项目搭建步骤

下载本课附带的 `cli_project.zip` 文件,解压到你偏好的开发目录。在项目文件夹里打开你的代码编辑器。

项目里包含一份完整的 README 文件,写明了搭建步骤。按以下步骤操作:

1. 把你的 Anthropic API key 加进 `.env` 文件
2. 用 UV(推荐)或 pip 安装依赖
3. 运行初始应用,确认一切正常

## 运行应用

在终端里进入你的项目目录,你会看到主要的项目文件,包括 `main.py`、`mcp_client.py` 和 `mcp_server.py`。

用以下任一命令启动应用:

```bash
# 如果用 UV(推荐)
uv run main.py

# 如果用标准 Python
python main.py
```

应用成功启动后,你会看到一个聊天提示符。可以问一个简单的问题(比如「1+1 等于几?」)来测试一下——你应该能很快收到 Claude 的回答。

基础搭建完成后,我们就可以开始实现 MCP 功能,探索客户端和服务器是如何通过 Model Control Protocol 通信的了。

---

对产品经理来说,「一个项目里同时做客户端和服务器只是为了教学」这句话值得记住:实际团队分工里,一个团队通常只专注做其中一端——比如对外提供服务的团队做 MCP 服务器,消费其他服务的产品团队做 MCP 客户端。规划人力和排期时,理清「我们这次到底是要造一个服务、还是要接入别人的服务」,能避免不必要的重复投入。
