# Building with the Claude API - 40 The full RAG flow 完整的 RAG 流程

> Course: Building with the Claude API · Lesson 40
> 课程: Building with the Claude API · 第 40 课
> 来源: Anthropic Academy (anthropic.skilljar.com),用户登录后提供英文原文,2026-07-17 整理翻译存档

我们已经讲完了 RAG、文本分块和文本嵌入的基础知识,现在来一步步走完完整的 RAG 流水线。这个例子会具体展示这些环节是如何配合在一起,检索出相关信息并生成回答的。

## 第一步:把源文本分块

首先,我们拿到源文档,把它拆成好管理的小块。这个例子里,我们用两个简单的文本段落:

- **第一节:医学研究** ——「今年我们对 XDR-47 有了重大认识突破,这是一种我们此前从未见过的『bug』。」
- **第二节:软件工程** ——「本部门投入了大量精力,研究我们分布式系统中的各类感染路径(infection vectors)。」

## 第二步:生成嵌入向量

接下来,我们用一个嵌入模型把每个文本小块转换成数值向量。为了方便理解,假设我们有一个「完美」的嵌入模型,它总是精确返回两个数字,而且我们知道每个数字代表什么。

在这个假想模型里:

- 第一个数字代表文本和「医学领域」的相关程度
- 第二个数字代表文本和「软件工程」的相关程度

对医学研究那一节,我们可能得到 `[0.97, 0.34]`——高度偏医学,但因为出现了「bug」这个词,也带一点软件相关的成分。对软件工程那一节,我们得到 `[0.30, 0.97]`——高度偏软件,但因为「感染路径」这个词,也带一点医学的意味。

### 归一化(Normalization)

嵌入 API 通常会做一步归一化处理,把每个向量的模长缩放到 1.0。这背后的数学原理你不需要操心——它是自动处理的。这样我们就得到了归一化后的向量,比如 `[0.944, 0.331]` 和 `[0.295, 0.955]`。

我们可以把这些嵌入向量画在一个单位圆上,每个点代表一个文本小块。

## 第三步:存入向量数据库

我们把这些嵌入向量存进一个向量数据库——一种专门为存储、比较、搜索这类长串数字(也就是我们的嵌入向量)而优化的数据库。

到这里,先暂停一下。目前所做的一切都是提前完成的预处理工作。现在,我们等待用户提交查询。

## 第四步:处理用户查询

当用户提出类似「我对这家公司很好奇,尤其想知道软件工程部门今年做了什么」这样的问题时,我们把他们的查询也送进同一个嵌入模型。

这条查询会被嵌入成类似 `[0.1, 0.89]` 的向量——医学得分低,软件工程得分高。归一化之后,得到 `[0.112, 0.993]`。

## 第五步:寻找相似的嵌入向量

我们把用户查询的嵌入向量发给向量数据库,让它找出最相似的已存储向量。

数据库会返回软件工程那一节,因为它和用户所问的内容最匹配。

## 相似度是如何计算的:余弦相似度(Cosine Similarity)

向量数据库用「余弦相似度」来判断哪些嵌入向量最相似。它衡量的是两个向量之间夹角的余弦值。

关于余弦相似度,有几个要点:

- 结果范围在 -1 到 1 之间
- 接近 1 表示高度相似
- 接近 -1 表示高度不同
- 0 表示两者垂直(没有关联)

在我们的例子里,用户查询和软件工程小块之间的余弦相似度是 0.983——非常高。而和医学研究小块的相似度只有 0.398——低得多。

### 余弦距离(Cosine Distance)

在向量数据库的文档里,你经常会看到「余弦距离」这个说法,计算方式很简单:`(1 - 余弦相似度)`。用余弦距离表示时:

- 接近 0 表示高度相似
- 数值越大表示相似度越低

这种换算方式,在很多场景下让数字更容易理解。

## 第六步:生成最终提示词

最后,我们把用户的问题和检索到的最相关文本小块组合起来,拼成一个提示词,发给 Claude 来生成回答。

提示词大概长这样:

```
Answer the user's question about the financial document.

<user_question>
How many bugs did engineers fix this year?
</user_question>

<report>
## Section 2: Software Engineering
This division dedicated significant effort to studying various infection vectors in our distributed systems
</report>
```

这样,完整的 RAG 流水线就走完了!系统根据语义相似度成功检索出最相关的信息,并把它作为上下文提供给 Claude,生成了准确的回答。

---

对产品经理来说,这一课把前几课的零散概念(分块、嵌入、检索)串成了一条完整的「用户提问 → 找答案依据 → 生成回复」流水线,这和一个智能客服系统的完整链路是一样的:先把知识库切块并「打好标签」入库(离线预处理),用户来问问题时,把问题也打上同样的标签,去库里找标签最接近的知识条目,再把这些条目喂给 AI 生成回答——理解这套完整链路,能帮你判断团队卡在哪个环节、该往哪个方向排查问题。
