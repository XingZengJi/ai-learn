# Building with the Claude API - 49 Rules of prompt caching 提示词缓存的规则

> Course: Building with the Claude API · Lesson 49
> 课程: Building with the Claude API · 第 49 课
> 来源: Anthropic Academy (anthropic.skilljar.com),用户登录后提供英文原文,2026-07-17 整理翻译存档

Claude 的提示词缓存,通过存储对你的消息所做的计算工作,让这些工作能在后续请求里被复用。这让后续请求既更快、又更省钱地执行——但前提是,你反复发送的是完全相同的内容。

整个流程很直接:你最初的请求会把处理工作写入缓存,后续请求则可以直接从缓存里读取,而不需要重新处理同样的内容。缓存的有效期是一小时,所以这项功能只有在你在这个时间窗口内反复发送相同内容时才有用。

## 缓存断点(Cache Breakpoints)

缓存不是自动开启的——你需要手动在消息里的特定块上添加「缓存断点」。运作方式如下:

- 对消息所做的处理工作,默认不会自动缓存
- 你必须手动在某个块上添加一个「缓存断点」
- 断点之前的所有内容对应的处理工作都会被缓存
- 只有当断点(含断点本身)之前的内容和之前完全一致时,后续请求才会用到缓存

要添加缓存断点,你需要用书写文本块的完整格式(longhand form),而不是简写形式。

简写形式没有地方能加上缓存控制字段,所以你必须使用展开的格式,把 `cache_control` 字段设为 `{"type": "ephemeral"}`。

## 缓存断点是如何运作的

当你在一条消息里放置一个缓存断点时,Claude 会缓存该断点(含断点本身)之前的所有处理工作。断点之后的内容,会按正常方式处理,不进行缓存。

要让缓存在后续请求里起作用,断点之前的内容必须完全一致。哪怕是加一个「please」这样的小改动,都会让缓存失效,迫使 Claude 重新处理全部内容。

## 跨消息的缓存

缓存断点可以横跨多条消息、多种消息类型。如果你把断点放在较靠后的一条消息里,那么之前所有的消息(用户消息、assistant 消息等)都会被包含进被缓存的内容里。

这对于「想缓存到某个节点为止的整段上下文」这类对话场景特别有用。

## 系统提示词和工具

你不只能在文本块上加缓存断点——它还可以加在:

- 系统提示词
- 工具定义
- 图像块
- 工具使用块和工具结果块

系统提示词和工具定义是缓存的绝佳候选,因为它们在请求之间很少变化。这往往是你从提示词缓存里获益最多的地方。

## 缓存的处理顺序

在幕后,Claude 会按特定顺序处理你请求里的各个组成部分:先是工具,然后是系统提示词,最后是消息。理解这个顺序,能帮你更有效地放置断点。

你总共最多可以添加四个缓存断点。比如,你可以缓存你的工具定义,再在对话历史中途加另一个断点。这样一来,当请求里不同部分各自发生变化时,你能灵活控制哪些内容被缓存。

## 最小内容长度

缓存有一个最低门槛:内容长度必须至少达到 1024 个 token 才能被缓存。这里指的是你试图缓存的所有消息和块加起来的总量,而不是单个块。

一句简单的「Hi there!」达不到这个门槛,但如果你把这段内容重复 500 次(或者你本来就有一个真正很长的提示词),它就会超过 1024 个 token,变得可以被缓存。

有效运用提示词缓存的关键,是识别出你的请求里哪些部分在多次调用之间保持不变,再有策略地放置断点,尽可能提高复用率,同时把缓存失效的风险降到最低。

---

对产品经理来说,「缓存断点」这个设计的关键教训是:缓存不是「开个开关就自动生效」的黑科技,而是需要你精心规划「哪部分内容基本不变、该缓存」「哪部分内容经常变、不该缓存」——就像仓储物流里把「常年不动的大件货物」放仓库深处、把「每天进出的零散货物」放门口一样,规划得好才能真正省成本、提速度,规划得不好(比如断点放错位置导致频繁失效)反而白费功夫。
