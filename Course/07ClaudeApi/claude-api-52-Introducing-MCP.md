Model Context Protocol(MCP,模型上下文协议)是一层通信协议,能让 Claude 获得上下文和工具,而不需要你自己写一堆繁琐的集成代码。可以把它理解为:把「定义工具、执行工具」这份负担,从你自己的服务器转移到专门的 MCP 服务器上。

初次接触 MCP 时,你会看到一些展示基础架构的示意图:一个 MCP 客户端(也就是你的服务器)连接到包含工具、提示词和资源的 MCP 服务器。每个 MCP 服务器都相当于某个外部服务的接口。

## 通过一个真实例子理解 MCP

假设你在搭建一个聊天界面,用户可以问 Claude 关于他们 GitHub 数据的问题。用户可能会问「我所有仓库里有哪些开放的 pull request?」。要回答这个问题,Claude 需要能访问 GitHub API 的工具。

没有 MCP 的话,你就得自己搭建所有 GitHub 集成的工具——这意味着,你想支持的每一项 GitHub 功能,都要自己写 schema 和函数。

## 工具函数带来的问题

GitHub 的功能非常庞大——仓库、pull request、issue、项目,还有更多。要搭建一个功能完整的 GitHub 聊天机器人,你需要写出数量惊人的工具。

每个工具都既需要一份 schema 定义,也需要一份函数实现。作为开发者,这意味着大量需要你自己编写、测试、维护的代码。

## MCP 是如何解决这个问题的

MCP 把「定义工具、执行工具」这份负担,从你的服务器转移到了 MCP 服务器上。你不需要自己写那些 GitHub 工具,它们是在一个专门的 MCP 服务器内部被编写和执行的。

MCP 服务器相当于对 GitHub 功能的一层封装,提供了一批现成的工具,你不需要自己实现就能直接使用。

MCP 服务器提供的是对外部服务所实现的数据或功能的访问入口。它们把复杂的集成工作打包成可复用的组件,任何应用都可以连接使用。

## 关于 MCP 的常见疑问

### 谁来编写 MCP 服务器?

任何人都可以创建一个 MCP 服务器的实现。通常,服务提供商自己也会推出官方的 MCP 实现。比如,AWS 可能会为它旗下的各种服务发布一个官方 MCP 服务器,里面包含相应的工具。

### MCP 和直接调用 API 有什么不同?

MCP 服务器已经为你提供好了工具 schema 和函数实现。如果你直接调用某个 API,你就得自己负责编写这些工具定义。而 MCP 帮你省下了这部分实现工作。

### MCP 不就是「工具使用」吗?

这是一个常见的误解。MCP 服务器和「工具使用」是相辅相成、但完全不同的概念。MCP 关注的是「谁来负责创建和维护这些工具」这件事——用了 MCP,别人已经替你写好了工具函数和 schema,它们被打包在 MCP 服务器内部。

核心洞察是:MCP 服务器为你提供了已经定义好的工具 schema 和函数,免去了你自己搭建和维护复杂集成的必要。

---

对产品经理来说,MCP 就像是「第三方开放平台」和「自建对接」的区别——你要接入微信支付,与其自己从零对接支付宝、银联、各家银行的接口,不如用一个成熟的支付网关,别人已经把这些复杂的集成都封装好了,你只管调用就行。MCP 做的正是这件事,只不过对象是「给 AI 用的工具」而不是「支付渠道」。
