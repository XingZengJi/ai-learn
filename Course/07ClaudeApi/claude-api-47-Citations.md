当 Claude 依据你提供的文档回答问题时,用户可能会以为它只是在凭训练数据作答。但如果 Claude 能准确指出「这条信息我是在哪里找到的」呢?这正是「引用(citations)」功能要解决的问题——它能让 Claude 引用源文档里的具体部分,向用户清楚展示每一条信息的出处。

## 为什么引用很重要

设想你问 Claude「地球的大气层是如何形成的」,得到了一份详细的回答。如果没有引用,用户没办法核实这条信息,也无从知道 Claude 实际上是在参考你提供的某份具体文档。引用功能解决了这个透明度问题,在 Claude 的回答和你的源材料之间建立起一条清晰的追溯路径。

## 开启引用功能

要开启引用,你需要调整文档消息的结构。给文档块加上两个新字段:

```python
{
    "type": "document",
    "source": {
        "type": "base64",
        "media_type": "application/pdf",
        "data": file_bytes,
    },
    "title": "earth.pdf",
    "citations": { "enabled": True }
}
```

`title` 字段给你的文档起一个可读的名字,而 `citations: {"enabled": True}` 告诉 Claude 要追踪它是从哪里找到相关信息的。

## 理解引用的结构

开启引用后,Claude 的响应会变得更复杂。你拿到的不再是简单的文本,而是包含每条论述引用信息的结构化数据。

每条引用包含以下几个关键信息:

- **cited_text** —— 文档里支撑 Claude 这条论述的确切原文
- **document_index** —— Claude 引用的是哪份文档(当你提供多份文档时很有用)
- **document_title** —— 你给这份文档起的标题
- **start_page_number** —— 引用文字的起始页码
- **end_page_number** —— 引用文字的结束页码

## 基于引用构建用户界面

引用功能真正的价值,在于用它来构建让这些信息触手可及的用户界面。你可以做出交互式的元素,让用户把鼠标悬停在引用标记上,就能看到信息具体来自哪里。

这能创造出一种透明的体验,让用户可以:

- 看到 Claude 的回答确实是基于真实的源材料
- 通过查看原始文档来核实信息
- 理解每条被引用信息周围的上下文

## 纯文本的引用

引用功能不仅限于 PDF 文档,也可以用在纯文本来源上。处理文本时,把文档结构改成这样:

```python
{
    "type": "document", 
    "source": {
        "type": "text",
        "media_type": "text/plain",
        "data": article_text,
    },
    "title": "earth_article",
    "citations": { "enabled": True }
}
```

对于纯文本来源,你拿到的不是页码,而是字符位置——精确定位到 Claude 在文本里找到某条信息的具体位置。

## 什么时候该用引用功能

在以下场景,引用功能特别有价值:

- 用户需要核实信息的准确性
- 你处理的是权威文档,用户应该能够引用查证
- 信息来源的透明度对你的应用至关重要
- 用户可能想深入探究某个具体事实周围的更多上下文

通过实现引用功能,你把 Claude 从一个「给出答案的黑盒」,变成了一个「展示论证过程」的透明研究助手。这能建立用户信任,也让他们在需要时能深入查阅你的源材料。

---

对产品经理来说,引用功能就是给 AI 的回答加上「脚注」——就像写研究报告或做尽调时,每个关键结论后面都要标注数据来源和页码,而不是让读者凭空相信结论。对于面向真实用户的产品(尤其涉及金融、医疗、法律这类需要「可验证」的场景),引用功能往往是从「demo 能跑」到「用户敢信」之间的关键一步。
