MCP 客户端是你的服务器和 MCP 服务器之间的通信桥梁。可以把它理解成你访问「MCP 服务器所提供的所有工具」的入口。当你需要使用外部工具或服务时,客户端会替你处理所有的消息传递和协议细节。

## 与传输方式无关的通信

MCP 的一大优势,是它对传输方式「不挑」(transport agnostic)——说白了就是,客户端和服务器可以用不同的通信方式互相对话。最常见的配置是,把 MCP 客户端和服务器都跑在同一台机器上,它们通过标准输入/输出(stdin/stdout)通信。

但你并不局限于这种方式。MCP 客户端和服务器也可以通过以下方式连接:

- HTTP
- WebSocket
- 其他各种网络协议

## 消息类型

一旦建立连接,客户端和服务器会交换 MCP 规范里定义的特定消息类型。你会用到的主要消息类型有:

- **ListToolsRequest / ListToolsResult**:客户端问服务器「你提供哪些工具?」,服务器返回一份可用工具列表
- **CallToolRequest / CallToolResult**:客户端要求服务器用特定参数执行某个工具,再拿到执行结果

## 完整流程示例

来看一个真实场景里,这些部件是如何协同工作的。假设用户问「我有哪些仓库?」——完整的通信流程是这样的:

流程从用户向你的服务器提交一个查询开始。你的服务器意识到,在向 Claude 发起请求之前,需要先给它提供一份可用工具的列表。

你的服务器向 MCP 客户端请求工具列表,MCP 客户端向 MCP 服务器发送一个 `ListToolsRequest`,收到一个 `ListToolsResult`。

现在,你的服务器已经具备了向 Claude 发起最初请求所需的一切——既有用户的问题,也有可用的工具列表。

Claude 检查这些工具,判断需要调用其中一个来回答问题,于是返回一个工具使用请求。

你的服务器让 MCP 客户端执行 Claude 请求的那个工具。MCP 客户端向 MCP 服务器发送一个 `CallToolRequest`,MCP 服务器随即向 GitHub 发起真正的请求。

GitHub 返回仓库数据,这份数据以 `CallToolResult` 的形式,经由 MCP 服务器流回,再到 MCP 客户端,最终到达你的服务器。

你的服务器把工具执行结果,以一条后续消息的形式发回给 Claude。此时 Claude 已经拥有了组织完整回答所需的全部信息。

最后,Claude 给出格式化好的答案,你的服务器把它传回给用户。

没错,这个流程涉及不少步骤,但每个组件的职责都很清晰。MCP 客户端把「与服务器通信」这部分复杂度抽象掉了,让你能专注于搭建应用本身的逻辑。等我们真正动手实现自己的 MCP 客户端和服务器时,你会看到这些部件在实践中是如何拼接在一起的。

---

对产品经理来说,这套「客户端问工具列表 → 转发给 Claude → Claude 决定调用哪个 → 客户端代为执行 → 结果回传」的流程,和公司里「前台助理帮你联系专业部门办事」很像:你不需要自己知道法务部、财务部各自的联系方式和流程,前台助理(MCP 客户端)会帮你转达需求、拿到结果再交给你——这层「中介」抽象掉了背后复杂的部门间协作细节。
