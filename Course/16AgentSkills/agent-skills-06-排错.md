# Introduction to Agent Skills - 06 Troubleshooting skills 排错

> Course: Introduction to Agent Skills(Agent Skills 入门)· Lesson 6
> 课程: Introduction to Agent Skills · 第 6 课 · 建议用时 15 分钟
> 来源: Anthropic Academy (anthropic.skilljar.com),登录后抓取英文原文翻译整理,2026-07-23

## What You'll Learn 本课目标

- 用 Skills 校验器在排错前先抓出结构问题
- 诊断并修复常见的触发与加载问题
- 解决企业/个人/项目/插件之间的优先级冲突
- 排查运行时错误: 缺依赖、权限、路径

Skills 出问题时,原因通常落在几个可预测的类别里: **不触发、不加载、有冲突、运行时失败**。好消息是大多数修法都很直接。

## 第一步: 先跑校验器

遇到问题**首先该试的是 agent skills 校验器命令**。安装步骤因操作系统而异,用 `uv` 是最快的方式。

装好之后,进到 Skill 目录里跑、或者在任何地方跑都行。**它能在你花时间排查别的东西之前,先抓出结构性问题。**

## 症状一: Skill 不触发

Skill 存在、校验也过了,但 Claude 该用的时候不用。**原因几乎总是 description。**

Claude 用的是**语义匹配**,你的请求得和描述的含义有重合。重合不够,就不匹配。做法:

- 拿描述对照你**实际**的提问方式
- 加上用户真的会说的触发短语
- 用不同说法测试,比如「帮我 profile 一下」「为什么这么慢?」「让它快点」
- **任何一种说法没触发,就把那些关键词加进描述**

## 症状二: Skill 不加载

如果你问 Claude「有哪些可用 Skills」时它没出现,检查这两条结构要求:

- **`SKILL.md` 必须在一个具名目录里面**,不能直接放在 skills 根目录下
- **文件名必须精确是 `SKILL.md`** —— "SKILL" 全大写,"md" 小写

跑 `claude --debug` 看加载错误,找提到你 Skill 名字的信息。**有时光这一步就能直接定位问题。**

## 症状三: 用错了 Skill

如果 Claude 用了错的 Skill、或者在几个 Skill 之间犯迷糊,**多半是你的描述太像了**。把它们写得更有区分度。

**描述写得尽量具体,不只是帮 Claude 决定何时用你的 Skill,也能防止和其他名字相近的 Skill 打架。**

## 症状四: 优先级冲突

如果你的个人 Skill 被无视了,可能有个企业级或更高优先级的同名 Skill。

![课程配图](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2Fa46l9irobhg0f5webscixp0bs%2Fpublic%2F1771527274%2FSkills6_12.1771527274623.png)

比如存在企业级 `code-review`,你又有个人版 `code-review`,**企业版每次都赢**。两个选择:

- **把你的 Skill 改成更独特的名字**(通常是更省事的路)
- 找管理员聊那个企业 Skill

## 症状五: 插件 Skills 不出现

装了插件但看不到它的 Skills?**清缓存、重启 Claude Code、重新安装。**

如果还是不出现,可能是插件结构有问题——**这时候校验器工具就真正体现价值了**。

## 症状六: 运行时错误

Skill 加载了,但执行中失败。常见原因:

- **缺依赖**: Skill 用到外部包就必须先装好。**把依赖信息写进 Skill 描述**,好让 Claude 知道需要什么
- **权限问题**: 脚本需要执行权限,对 Skill 引用的脚本跑 `chmod +x`
- **路径分隔符**: **一律用正斜杠,Windows 上也是**

## 快速排错清单

| 症状 | 检查 |
|---|---|
| 不触发 | 改进描述,加触发短语 |
| 不加载 | 检查路径、文件名、YAML 语法 |
| 用错 Skill | 让各描述之间区分度更高 |
| 被覆盖 | 检查优先级层次,必要时改名 |
| 插件 Skills 不见 | 清缓存并重装 |
| 运行时失败 | 检查依赖、权限、路径 |

## 课程小结

课程收尾的一句话值得记住:

> **最好的 Skills 来自真实的痛点——从你发现自己最常重复的那些指令开始。**

## 对产品经理来说

这一课的结构本身就是个好东西: **按症状分类的排错表**。注意它的排序——**先跑校验器,再排查其他**。这是很成熟的排错思维: **先用自动化工具排除掉一整类可能性,再动脑子**。人的直觉倾向于直接猜最可能的原因,但那往往比跑一个命令慢得多。

六个症状里有三个的根因是**同一个东西——描述写得不好**(不触发、用错 Skill,以及部分优先级问题)。这跟第 2 课那条呼应上了: **`description` 是 Claude 唯一能看到的选择依据**。放到更普遍的语境,这是"检索层"的问题:**内容再好,如果索引不准,就等于不存在。**

「**把依赖信息写进 Skill 描述,好让 Claude 知道需要什么**」这条小技巧背后有个通用原则: **让系统的前置条件对使用者可见**。这在写文档、设计 API、做交接时都成立——**沉默的前置依赖是故障的常见来源**,因为它只在出错时才第一次暴露。
